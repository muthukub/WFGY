If the [Community Fix Lab](../README.md) is the broader entry page for community repair assets, this folder is the reproduction-pack lane for portable case bundles, rerun-ready mini kits, and compact troubleshooting packages that others can inspect or reuse with minimal friction. 🧭

Use this folder when a contribution is mainly about **packaging one case, one route, or one repair demonstration into a portable bundle**, not when the contribution is mainly a notebook, JSON fixture, prompt pack, workflow recipe, or official demo.

Short version:

> official layer gives the repair grammar  
> this folder helps package that grammar into portable reusable bundles

---

## Quick start 🚀

### I want to contribute a reproduction pack

Use this path:

1. decide whether the asset is really a portable pack contribution
2. route the case first with the atlas
3. keep the pack scoped to one family, one case, or one troubleshooting slice
4. make the included files and usage steps explicit
5. explain the expected result, limits, and warnings

### I want to browse reproduction packs

Use this path:

1. open one pack with a clear case or family scope
2. inspect the short README and included files
3. identify the route or repair idea being packaged
4. check how to run or inspect the pack
5. check the expected result and limitations

Short version:

> route first  
> keep the pack compact  
> make reuse easy ✨

## What belongs here ✅

Good reproduction packs include:

- one compact case bundle
- one reproducible route-and-repair kit
- one small teaching bundle for a boundary case
- one portable rerun pack
- one notebook-plus-fixture mini package

A good pack should be:

- clearly named
- easy to inspect
- small enough to reuse
- explicit about included files
- explicit about expected result

---

## What does not belong here 🚫

Please do not use this folder for:

- giant archive-like dumps
- undocumented asset bundles
- unclear collections of unrelated files
- packs with no route context
- packs that claim official status without review
- bundles that hide critical setup steps

A reproduction pack should make one troubleshooting path easier to share and rerun, not turn the folder into a mystery archive.

---

## Suggested pack pattern 🧩

A useful reproduction pack usually includes:

1. one short README
2. case description
3. included files list
4. how to run or inspect it
5. expected result
6. limits or warnings

That is enough to make the pack useful for others.

---

## Suggested naming style 📌

Examples:

- `f1-grounding-reproduction-pack-v1/`
- `f4-execution-closure-reproduction-pack-v1/`
- `f7-container-fidelity-reproduction-pack-v1/`

Use folders when multiple files are required.

Keep names readable and compact.

---

## What a good first pack looks like 🌱

A strong first contribution usually looks like this:

- one family or one case
- one portable bundle
- one short README
- one clear file list
- one expected outcome
- one short note on limits

Small, clean packs are much better than giant messy bundles.

## Review standard ✅

A reproduction-pack contribution is much more likely to be accepted if it is:

- clearly named
- easy to inspect
- easy to rerun or reuse
- connected to atlas routing
- explicit about included files
- honest about limitations

Messy bundles are still messy.  
Clean compact packs are more valuable.
